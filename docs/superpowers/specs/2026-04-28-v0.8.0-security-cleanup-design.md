# v0.8.0 Security Cleanup — Design Spec

## Context

BarePitch is a plain PHP/MySQL soccer team management app. This milestone is a security hardening pass before v1.0.0. The app has a solid baseline (CSRF on every POST, prepared statements, `e()` on all output, timing-safe login) but several gaps remain.

Production is HTTPS-only.

---

## Goals

- Fix the session timeout bypass in `requireLogin()`
- Add `cookie_secure` to session configuration
- Send HTTP security headers on every request
- Add database-backed login rate limiting
- Rotate CSRF token after every validated POST
- Add `public/.htaccess` for directory listing and HTTP→HTTPS redirect
- i18n carry-over: replace hardcoded "Upcoming"/"Past" headings in matches view

---

## Section 1: Session security

### `cookie_secure` flag

Added to `session_start()` options in `app/config/bootstrap.php`:

```php
'cookie_secure' => str_starts_with(getenv('APP_URL') ?: '', 'https://'),
```

Conditional on `APP_URL` — works for local HTTP development and production HTTPS without any environment change.

### Timeout enforcement in `requireLogin()`

**Current bug:** `requireLogin()` in `app/helpers/auth.php` only checks `$_SESSION['user_id']`. The 7200s inactivity timeout is only enforced in `AuthService::isLoggedIn()`, which is called only on `login.php`. All other protected pages bypass the timeout entirely.

**Fix:** Inline the timeout check inside `requireLogin()`:

```php
function requireLogin(): void
{
    if (!isset($_SESSION['user_id'], $_SESSION['last_activity'])) {
        header('Location: /login.php');
        exit;
    }

    if (time() - $_SESSION['last_activity'] > 7200) {
        $_SESSION = [];
        session_destroy();
        header('Location: /login.php');
        exit;
    }

    $_SESSION['last_activity'] = time();
}
```

`AuthService::isLoggedIn()` keeps its own timeout check (used on the login page redirect) — no change there.

## Section 2: HTTP security headers

A `sendSecurityHeaders()` function added to `app/helpers/auth.php`, called once in `bootstrap.php` after `session_start()`.

| Header | Value |
|--------|-------|
| `X-Content-Type-Options` | `nosniff` |
| `X-Frame-Options` | `DENY` |
| `Referrer-Policy` | `strict-origin-when-cross-origin` |
| `Content-Security-Policy` | `default-src 'self'; script-src 'self' 'unsafe-inline'; style-src 'self' 'unsafe-inline'` |
| `Strict-Transport-Security` | `max-age=31536000; includeSubDomains` — only when `APP_URL` starts with `https://` |

`unsafe-inline` is required for the inline `<script>` in `footer.php` and inline `style` attributes still present in some views.

## Section 3: Login rate limiting

### Database migration

New file: `database/migrations/006_create_login_attempts_table.sql`

```sql
CREATE TABLE login_attempts (
    id           INT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    ip_address   VARCHAR(45)  NOT NULL,
    attempted_at DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP,
    INDEX idx_ip_time (ip_address, attempted_at)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

### AuthService changes

Three new methods in `app/services/AuthService.php`:

- `isRateLimited(string $ip): bool` — counts rows for this IP in the last 15 minutes; returns `true` if ≥ 10
- `recordFailedAttempt(string $ip): void` — inserts a row on failed login
- `clearAttempts(string $ip): void` — deletes all rows for this IP on successful login

Probabilistic cleanup (1-in-100 chance per attempt check) deletes rows older than 1 hour to keep the table lean without a cron job.

### Controller change (`public/login.php`)

IP address read from `$_SERVER['REMOTE_ADDR']`. Rate limit checked before any credential validation. Shows the same generic error message on lockout ("Invalid email or password.") — no information about the rate limit is revealed.

## Section 4: CSRF token rotation

New function in `app/helpers/csrf.php`:

```php
function rotateCsrfToken(): void
{
    unset($_SESSION['csrf_token']);
}
```

Called immediately after `validateCsrf()` succeeds in every POST-handling controller, before any redirect or render:

- `public/login.php`
- `public/logout.php`
- `public/team_create.php`
- `public/team_edit.php`
- `public/player_create.php`
- `public/player_edit.php`
- `public/match_create.php`
- `public/match_edit.php`
- `public/match.php`

**Edge case:** Browser back button after a POST presents a stale token. The existing 403 response is unchanged — no new user-visible error needed.

## Section 5: .htaccess

New file: `public/.htaccess`

```apache
# Disable directory listing
Options -Indexes

# Deny PHP execution in assets directory
<Directory "assets">
    <FilesMatch "\.php$">
        Require all denied
    </FilesMatch>
</Directory>

# Redirect HTTP to HTTPS (skip localhost)
<IfModule mod_rewrite.c>
    RewriteEngine On
    RewriteCond %{HTTPS} off
    RewriteCond %{REMOTE_ADDR} !^127\.0\.0\.1$
    RewriteCond %{REMOTE_ADDR} !^::1$
    RewriteRule ^ https://%{HTTP_HOST}%{REQUEST_URI} [L,R=301]
</IfModule>
```

Requires Apache `mod_rewrite`. On Nginx the HTTP→HTTPS redirect is handled in the server block — this file is a no-op there.
